## 1. 产品愿景与定位

HRMS 旨在以 SAP HCM 与 Oracle PeopleSoft HCM 的经典模块结构为蓝本，交付一套**可演示的 MVP 原型**——覆盖"组织 / 员工 / 考勤假期 / 薪酬 / 招聘 / 绩效 / 自助"七域端到端主路径，技术上验证多关系数据库适配能力（PostgreSQL / MySQL 优先），架构上为后续向正式产品演进留出模块化扩展口。MVP 不追求合规性、不承诺规模化部署，只承诺"在演示规模下跑通主流程，并能在 PG↔MySQL 间无缝切换"。

## 4. MVP 范围（In-Scope）

MVP 共 6 个 Epic，均以"演示主路径可跑通"为最低交付门槛：

1. **Epic 01 组织与员工主数据**——组织树、员工档案、入离调转
2. **Epic 02 考勤与假期**——班次/排班/打卡录入、请假/加班审批、余额年度结算
3. **Epic 03 薪酬基础**——薪资档案、月度工资 run（一档示例）、工资条
4. **Epic 04 招聘与入职**——职位、候选人、面试评价、Offer→员工档案草稿
5. **Epic 05 绩效**——周期/模板/目标录入/自评/上级评分/最终得分
6. **Epic 06 ESS/MSS 自助服务**——员工端 + 经理端的自助查询与待办审批

横切能力（统一在第 7 章定义）：登录认证、RBAC、组织数据权限、审批流、审计日志、附件、字典。

## 5. 非目标（Out-of-Scope）—— 完整清单

下列项**MVP 一律不做**，进入 Roadmap（第 10 章）。所有项均为 PM 已拍板决策：

### 5.1 业务范围类
- ❌ 多公司 / 多法人 / 集团合并报表（MVP 仅 1 家公司，根节点固化为常量种子）
- ❌ 多语言 / 国际派遣 / 海外薪酬
- ❌ 多币种 / 多时区
- ❌ Talent Succession Pipeline（继任者计划）
- ❌ Learning Management（学习与培训）
- ❌ 复杂福利计划（DC/DB Plans，待 Roadmap）
- ❌ Travel Management（差旅）
- ❌ 360 度环评、同事互评、跨部门评价
- ❌ 申诉流程（绩效结果不允许申诉，进 backlog）
- ❌ 调薪历史回溯与版本化
- ❌ 跨部门外派、矩阵汇报关系

### 5.2 薪酬简化类
- ❌ 真实税务全国规则（仅做"北京 2024 一档示例"，非合规口径）
- ❌ 年度个税汇算清缴
- ❌ 专项附加扣除（子女教育/房贷/赡养老人等）
- ❌ 入职/离职当月按日折算
- ❌ 跨月补发补扣
- ❌ 缺勤联动扣薪（MVP 缺勤仅记录、不扣薪）
- ❌ 加班联动工资（MVP 加班仅记录小时数）
- ❌ 多套薪资方案（仅一套全公司统一）
- ❌ 银企直连发薪、金税系统申报对接

### 5.3 流程能力简化
- ❌ 会签（多人并行同节点）
- ❌ 分支判断节点
- ❌ 子流程嵌套
- ❌ 代理人 / 委托审批
- ❌ 超过 3 级的串行审批
- ❌ 审批人离职后的复杂改派（仅做角色顺延 + Admin 兜底）

### 5.4 集成与渠道
- ❌ 钉钉 / 企微 / 飞书 SSO
- ❌ 与外部 SAP / Oracle / 金蝶系统真实对接
- ❌ 招聘渠道集成（智联、BOSS、领英等）
- ❌ 移动 App（仅响应式 Web，不做原生）
- ❌ 与 OA / CRM 单据打通

### 5.5 状态机与可逆性
- ❌ 入离调转状态变更可逆（误操作只能新建反向变更单）
- ❌ 工资 run 直接编辑（必须红冲后重算）

### 5.6 运维与扩展
- ❌ SaaS 多租户
- ❌ 微服务拆分（MVP 单体部署）
- ❌ 集群高可用与灰度
- ❌ 演示数据重置改为生产备份恢复机制

## 6. Epic 详述

### Epic 01 组织与员工主数据（Organization & Employee Master Data）

**对标**：SAP OM+PA / PeopleSoft Workforce Admin

**业务目标**
建立单公司下的多级组织树（公司→部门→岗位→职位）、员工档案完整生命周期（入职/调岗/晋升/离职），并保证状态变更全程留痕，支撑后续所有 Epic 的人员基础数据。

**关键用户故事**
1. As an **Admin**, I want 创建并维护多级部门树（最深 5 级），so that 反映真实组织结构。
2. As an **Admin**, I want 在部门下创建岗位（含编制数、职位关联、汇报关系），so that 描述编制蓝图。
3. As an **HR Manager**, I want 录入完整员工档案（基本信息/合同/教育/工作经历/紧急联系人/银行卡/家属），so that 后续薪酬考勤可调用。
4. As an **HR Manager**, I want 为新员工执行"入职"操作（绑定岗位+生效日），so that 员工状态变为 active 并自动建系统账号。
5. As an **HR Manager**, I want 提交调岗/晋升单，so that 经审批后员工任职关系更新且历史保留。
6. As an **HR Manager**, I want 执行"离职"操作（含末日/原因/工作交接），so that 员工置为 terminated 且账号自动禁用。
7. As a **Line Manager**, I want 查看本部门及所有递归下属部门员工列表，so that 进行人员管理。
8. As an **Admin**, I want 对所有人员变更（入离调转、字段编辑）查到审计日志，so that 满足合规追溯。

**主要数据对象**
公司、部门、岗位、职位、员工、合同、教育经历、工作经历、紧急联系人、银行卡、家属、任职关系、人员变更单（HireForm / TransferForm / TerminationForm）、审计日志。

**核心业务流程**
1. Admin 创建部门→岗位→职位
2. HR Manager 发起入职单（关联岗位+生效日+合同信息）→ 1 级审批 → 系统自动建账号 → 员工状态 active
3. HR Manager 发起调岗/晋升单 → 1～2 级审批 → 任职关系结束旧记录、新建新记录、原任职 end_date=新生效日-1
4. HR Manager 发起离职单 → 1 级审批 → 末日生效后账号禁用、相关待办关闭、保留历史

**验收标准（AC）**
1. 创建部门时，父部门必填且不能选自身或自身后代，否则报错"循环引用"。
2. 创建员工时，身份证号若已存在历史员工记录（含离职），系统**禁止**直接重复创建，提示"已存在 [姓名]，请走重新雇佣流程或联系 Admin 合并"。
3. 入职单审批通过后 5 秒内系统自动创建账号，初始密码为身份证后 6 位且首次登录强制改密。
4. 调岗单生效后，旧任职记录 end_date 自动等于新生效日前一天，且员工只允许有 1 条 active 任职。
5. 离职单生效后，员工账号置为 disabled、所有待办自动关闭、其他员工不可再选其为审批人。
6. Line Manager 查询员工列表时，能看到本部门及递归子树所有部门员工，跨部门员工不可见。
7. 删除部门时若仍存在 active 员工或 active 子部门，系统拒绝并提示需先迁移。
8. 入离调转单一旦生效**不可逆**，误操作只能新建反向单（如"反向离职"=重新雇佣），系统提示且记录关联单号。
9. 所有人员主数据字段编辑均产生动作级审计日志（who/when/what/result）。
10. 身份证、银行卡、手机号在列表页脱敏显示（如 110**********1234），详情页 Admin/HR Manager 可见明文。

**与其他 Epic 的依赖**：被所有其他 Epic 依赖；Epic 04 招聘的"Offer 接受"会自动调用本 Epic 的"创建员工草稿"。

**Epic 内的硬边界声明**
- 不做多公司/多法人合并；根节点 1 家公司常量化
- 不做调薪历史与版本化（薪资变化通过 Epic 03 的薪资档案直接覆盖）
- 不做矩阵汇报、跨部门外派
- 不做状态可逆（误操作走反向单）
- 不做按日折算（入离调转生效日仅影响后续月份计算）
- 不做家属信息的独立审批流（家属字段编辑直接生效）

### Epic 02 考勤与假期（Attendance & Leave）

**对标**：SAP PT / PeopleSoft Time and Labor + Absence Management

**业务目标**
管理员工的班次、排班、考勤记录与请假申请，提供假期余额查询与年度结算，作为后续薪酬计算（MVP 中不联动）与日常 HR 管理的基础。

**关键用户故事**
1. As an **Admin**, I want 维护班次模板（如朝九晚六/弹性班/夜班），so that 可指派给不同员工。
2. As an **HR Manager**, I want 按月为部门或员工指派班次（排班），so that 后续考勤有基线。
3. As an **HR Manager**, I want 通过 CSV 批量导入打卡记录（员工编号/日期/上下班时间），so that 在没有硬件接入下补录考勤。
4. As an **Employee**, I want 提交请假单（假类/起止日/时长/原因/附件），so that 走审批流。
5. As a **Line Manager**, I want 在 MSS 待办中审批本部门请假单，so that 同意或驳回并写明意见。
6. As an **Employee**, I want 查询自己各假类的余额与历史记录，so that 知道还能请几天。
7. As an **Employee**, I want 提交加班申请单（仅记录小时数，不联动工资），so that 月底统计加班时长。
8. As an **Admin**, I want 在年初/年末执行假期年度结算（年假结转上限 5 天 + 3 月底前用完，事假/病假不结转，调休 2 月内有效），so that 自动调整余额。

**主要数据对象**
班次、排班、打卡记录、考勤日汇总、假类（年假/事假/病假/婚假/产假/陪产假/丧假/调休）、请假单、加班单、假期余额、假期年度结算批次。

**核心业务流程**
1. Admin 维护班次 → HR Manager 排班（按部门/员工）
2. 打卡录入：手工或 CSV 导入 → 系统按排班生成考勤日汇总（出勤/缺勤/迟到/早退）
3. 员工提交请假单 → 审批流（1～3 级，依假类与天数配置） → 通过后扣减余额
4. 员工提交加班单 → Line Manager 审批 → 加班小时数累计（不进入工资）
5. 年度结算：Admin 触发批次任务，按规则结转/清零/计提次年余额

**验收标准**
1. 创建请假单时，若起止日跨越员工离职日，系统拒绝并提示"超出在职日期"。
2. 请假天数计算按"自然日"而非"工作日"（MVP 简化），半天单位支持 0.5。
3. 余额不足时禁止提交且明确提示，不允许出现负值。
4. 同一员工同一时段不允许重复请假（与已批准请假单时间区间冲突时拒绝）。
5. 请假单审批通过即扣减余额；驳回不扣减；撤销已批准单需在生效日前由发起人发起、HR Manager 二次确认才能恢复余额。
6. 跨月请假单（如 1/30→2/3）必须按整段一次审批，不可拆月。
7. 加班单批准后仅记录 hours 字段，**不**触发工资计算。
8. 调休假申请需选择来源加班单（FIFO 抵扣），调休 2 月内未用完则自动失效。
9. 年度结算执行后，年假结转量超过 5 天的部分被清零；调休余额按 2 月有效窗口截断；事假/病假当年清零。
10. 打卡 CSV 导入失败时（格式错/工号不存在/时间倒挂）输出错误行号与原因，且整批不写入。

**与其他 Epic 的依赖**：依赖 Epic 01 的员工主数据；MVP 中**不**反向影响 Epic 03 工资。

**Epic 内的硬边界声明**
- 不接入硬件考勤机，仅手工/CSV
- 不做工时制差异（标准/不定时/综合 MVP 全部按标准制）
- 不做加班自动联动工资
- 不做缺勤自动扣薪
- 不做请假流程的会签或代理人
- 不做按工作日扣减（仅自然日）

### Epic 03 薪酬基础（Payroll Basic）

**对标**：SAP PY / PeopleSoft Payroll（**严格简化版**）

**业务目标**
为每月在职员工执行一次性的工资计算 run，输出工资条；支持工资 run 锁定与红冲；个税与社保公积金均使用单一示例口径，**仅作演示，不承诺合规**。

**关键用户故事**
1. As an **HR Manager**, I want 维护员工薪资档案（基本工资/岗位工资/绩效基数/补贴），so that 工资 run 可读取。
2. As a **Payroll Specialist**, I want 创建月度工资周期（如 2026-06）并锁定参与员工范围（全月在职），so that 准备结算。
3. As a **Payroll Specialist**, I want 执行工资 run（应发→个税→社保公积金→实发），so that 自动生成所有员工本期工资明细。
4. As a **Payroll Specialist**, I want 在 run 完成后查看个人/全员汇总，so that 复核结果。
5. As a **Payroll Specialist**, I want 一键锁定 run，锁定后任何明细都不可改，so that 保证数据完整性。
6. As a **Payroll Specialist**, I want 对已锁定 run 发起"红冲"（生成等额负向 run）并重算，so that 修正错误而非直接编辑。
7. As an **Employee**, I want 在 ESS 中查看本人本期工资条，so that 知晓收入构成。
8. As an **Admin**, I want 看到所有薪资字段的修改前后值审计，so that 满足合规审计。

**主要数据对象**
薪资档案、薪资项目（基本工资/岗位工资/绩效/补贴/社保扣款/公积金扣款/个税/实发）、工资周期、工资 run、工资明细、工资条、累计预扣预缴台账、社保公积金费率常量表（北京 2024 一档示例）、红冲单。

**核心业务流程**
1. HR Manager 维护员工薪资档案（生效日 + 各项金额）
2. Payroll Specialist 创建工资周期（YYYY-MM）并选定 run 类型（正常/红冲）
3. 系统计算：
   - 应发 = 基本工资 + 岗位工资 + 绩效 + 补贴
   - 社保扣款 = 应发 × 个人社保费率（示例固定费率）
   - 公积金扣款 = 应发 × 个人公积金费率（示例固定）
   - 累计应纳税所得额 = 累计应发 - 累计社保 - 累计公积金 - 累计起征点（5000×已过月份）
   - 个税 = 按累计预扣预缴档位查表 - 已扣个税
   - 实发 = 应发 - 社保 - 公积金 - 个税
4. Payroll Specialist 复核 → 锁定 run → 工资条对员工 ESS 可见
5. 如需修正：发起红冲单（生成原 run 的负向镜像）→ 再发起新 run

**验收标准**
1. 工资 run 仅纳入"全月在职"员工：本月内有入职或离职动作的员工**不在**本 run 内（演示口径明示）。
2. 计算结果中实发金额不允许为负，若计算出负值则该员工本期 run 失败并在汇总中列入异常清单。
3. 工资 run 一旦锁定，所有薪资明细字段只读；尝试编辑接口直接返回 403。
4. 红冲单生成的负向 run 金额必须与原 run 完全等额相反；红冲后必须存在新的正向 run 才允许下一周期发起。
5. 个税计算结果与"累计预扣预缴台账"一致：第 N 月个税 = 累计应纳税额对应税额 - 前 N-1 月已扣总额。
6. 同一员工同一周期不允许两个 active 正向 run。
7. 工资条对员工本人在 ESS 中可见，对其他员工不可见，对 Admin 可见明文且字段级审计可查前后值。
8. 演示规模（20 员工）下，工资 run 整体执行时间 ≤ 30 秒。
9. 缺勤、加班数据**不**进入本期工资（边界明示）。
10. 入职/离职当月员工不参与 run（边界明示），相关说明在汇总页提示。

**与其他 Epic 的依赖**：依赖 Epic 01 员工主数据；可读 Epic 02 数据但不联动；被 Epic 06 工资条查询调用。

**Epic 内的硬边界声明**
- 不做按日折算
- 不做跨月补发补扣
- 不做专项附加扣除
- 不做年度汇算
- 不做多套薪资方案
- 不做考勤/加班联动
- 不允许直接编辑已锁定 run，只能红冲重算

### Epic 04 招聘与入职（Recruitment & Onboarding）

**对标**：SAP Recruitment / PeopleSoft eRecruit

**业务目标**
覆盖职位发布→候选人简历→面试评价→Offer 发放→自动转员工档案草稿的招聘漏斗，与 Epic 01 入职流程无缝衔接。

**关键用户故事**
1. As a **Recruiter**, I want 发布职位（关联岗位+JD+招聘人数+截止日），so that 候选人可投递。
2. As a **Recruiter**, I want 录入候选人简历（基础信息/教育/工作/期望薪资/简历附件），so that 进入候选库。
3. As a **Recruiter**, I want 安排面试（轮次/面试官/时间/地点），so that 通知面试官。
4. As a **Line Manager**, I want 在面试后录入评价（评分/通过/不通过/备注），so that 推动招聘下一步。
5. As a **Recruiter**, I want 对通过候选人发起 Offer（含薪资/入职日/岗位），so that 由 HR Manager 审批。
6. As a **HR Manager**, I want 审批 Offer，so that 把控成本与编制。
7. As a **Recruiter**, I want 在系统标记候选人接受/拒绝 Offer，so that 自动驱动下一步。
8. As a **Recruiter**, I want 候选人接受 Offer 后系统自动创建员工档案草稿（status=pending_onboard），so that HR 在入职日确认转 active。

**主要数据对象**
职位需求、候选人、简历附件、面试安排、面试评价、Offer、Offer 审批历史、候选人状态历史。

**核心业务流程**
1. Recruiter 创建职位需求 → 选关联岗位 → 设招聘人数与截止日
2. Recruiter 录入候选人 → 投递关联职位
3. 面试编排：N 轮面试官 → 录入评价 → 流转到下一轮或淘汰
4. 通过候选人 → Recruiter 发起 Offer → HR Manager 审批
5. Offer 发出 → 候选人接受/拒绝（系统手动标记） → 接受则自动建员工档案草稿 → HR 在入职日点"确认入职"转 active

**验收标准**
1. 候选人身份证号若与历史员工记录（含离职）匹配，系统提示"该身份证已存在员工档案 [姓名]，是否走重新雇佣流程"，禁止直接重复创建。
2. 面试时间冲突（同面试官同时段）时系统提示但不强制阻塞，由 Recruiter 决定。
3. Offer 一经审批通过即不可改金额；如需调整必须撤销原 Offer 后重发。
4. 候选人接受 Offer 后 5 秒内系统自动建员工档案草稿，员工编号此时分配。
5. 草稿员工档案 status=pending_onboard，**不**进入工资 run、**不**进入排班。
6. HR 在入职日（含当日）点"确认入职"，员工档案 status=active 并触发 Epic 01 的账号自动创建流程。
7. 候选人若拒绝或超期未接受（默认 7 天），Offer 自动作废，候选人状态回 offer_declined。
8. 简历附件大小 ≤10MB，格式限 pdf/docx/jpg/png。
9. 一个职位达到招聘人数后状态自动置为 closed，不再接收新候选人投递。
10. 面试评价历史保留所有版本，不允许覆盖编辑。

**与其他 Epic 的依赖**：Epic 01 员工档案为下游；横切的审批流支撑 Offer 审批。

**Epic 内的硬边界声明**
- 不做内推与社交分享
- 不做招聘渠道集成
- 不做候选人前台门户/自助投递
- 不做 AI 简历筛选
- 不做背调流程
- Offer 审批最多 1 级（HR Manager），不做更长链路

### Epic 05 绩效（Performance）

**对标**：SAP PM / PeopleSoft ePerformance（**MVP 骨架**）

**业务目标**
建立周期化绩效考核闭环：模板→目标→自评→上级评分→最终得分，作为后续 Roadmap 360/校准/激励的骨架。

**关键用户故事**
1. As an **HR Manager**, I want 创建绩效周期（如 2026 H1）与适用范围（部门/全员），so that 启动考核。
2. As an **HR Manager**, I want 配置绩效模板（评分项/权重/分值范围），so that 全公司或按部门套用。
3. As an **Employee**, I want 在周期初录入个人目标（KPI/OKR 简化版，文本+权重），so that 上级确认。
4. As a **Line Manager**, I want 确认下属目标，so that 进入考核期。
5. As an **Employee**, I want 在周期末做自评（按评分项打分+评论），so that 上级有参考。
6. As a **Line Manager**, I want 给下属打上级评分，so that 计算最终得分。
7. As an **HR Manager**, I want 看到全公司绩效汇总，so that 输出报表。
8. As an **Employee**, I want 看到自己最终得分，so that 了解结果（**MVP 不支持申诉**）。

**主要数据对象**
绩效周期、绩效模板、评分项、员工绩效单、目标条目、自评、上级评分、最终得分。

**核心业务流程**
1. HR Manager 建周期 → 选模板 → 选适用员工范围 → 系统生成每人 1 份绩效单
2. 员工录入目标 → 上级确认
3. 周期末：员工填自评 → 提交给上级
4. 上级填上级评分 → 提交
5. 系统按公式：**最终得分 = 自评 × 0.3 + 上级 × 0.7** 计算并锁定，对员工可见

**验收标准**
1. 同一员工同一周期只能有 1 份绩效单。
2. 上级评分提交后即触发最终得分自动计算并锁定，所有评分字段不可再改。
3. 若员工在周期内离职，绩效单状态置为 terminated，最终得分不计算。
4. 若员工只完成自评、上级未评分，则周期截止后系统将该单标记为 incomplete，不计算最终得分。
5. 自评和上级评分的分值必须落在模板配置的区间（如 1-5 分），越界拒绝提交。
6. 绩效模板锁定后不可改评分项和权重；如需调整必须新版本号 + 不影响已使用的旧周期。
7. Line Manager 仅能看到本部门及递归下属员工的绩效单，跨部门不可见。
8. 最终得分对员工本人在 ESS 中可见；员工**不能**对结果发起申诉（MVP 边界明示）。
9. 绩效单状态机：draft → goal_set → in_appraisal → completed/terminated/incomplete，状态流转不可逆。
10. HR Manager 全公司绩效汇总报表查询响应时间 ≤3 秒（演示规模）。

**与其他 Epic 的依赖**：依赖 Epic 01 员工主数据 + 部门递归权限；被 Epic 06 ESS 调用展示。

**Epic 内的硬边界声明**
- 不做申诉流程
- 不做 360 评价
- 不做绩效校准会议
- 不做绩效与薪酬激励联动
- 不做 OKR 复杂结构（仅文本+权重）
- 不做绩效结果版本化

### Epic 06 ESS/MSS 自助服务（Employee & Manager Self Service）

**对标**：SAP ESS/MSS / PeopleSoft Self Service

**业务目标**
为员工和经理提供一致的自助门户：员工自助查询个人数据与发起单据；经理自助处理待办与查看团队信息。

**关键用户故事**
1. As an **Employee**, I want 登录后看到 ESS 首页（个人卡片/待办/最新工资条入口/请假入口），so that 一站式入口。
2. As an **Employee**, I want 查看与编辑自己的紧急联系人/银行卡/家属（基本字段不可改），so that 自助维护。
3. As an **Employee**, I want 在 ESS 中提请假/加班单并跟踪审批进度，so that 不依赖 HR。
4. As an **Employee**, I want 查询本人历史工资条（已锁定 run 才可见），so that 自助下载。
5. As an **Employee**, I want 在 ESS 中做绩效自评/录入目标，so that 完成考核任务。
6. As a **Line Manager**, I want 在 MSS 看到统一待办列表（请假/加班/调岗等），so that 集中处理。
7. As a **Line Manager**, I want 在 MSS 看到本部门递归下属员工列表与基础信息，so that 团队管理。
8. As a **Line Manager**, I want 在 MSS 中做下属绩效上级评分，so that 完成考核。

**主要数据对象**
（无独立数据对象；聚合 Epic 01-05 的数据通过权限过滤展示）

**核心业务流程**
1. 登录 → 按角色路由到 ESS 或 MSS 首页
2. ESS：个人信息 / 假期 / 工资条 / 绩效 四大菜单
3. MSS：待办 / 团队 / 绩效评分 三大菜单
4. 单据提交 → 后端走横切审批流 → 待办推送给审批人 → 审批结果回流到发起人

**验收标准**
1. ESS 首页加载时间 ≤2 秒（演示规模）。
2. 员工只能编辑紧急联系人、银行卡、家属字段；身份证、入职日、岗位、薪资等字段只读。
3. 员工修改银行卡需二次输入登录密码确认。
4. ESS 工资条仅展示状态为 locked 的工资 run，未锁定 run 不可见。
5. ESS 工资条详情页对身份证号、银行卡号脱敏展示。
6. MSS 待办列表按时间倒序，显示标题/类型/发起人/天数；点击进入对应单据详情。
7. MSS 团队页只展示本部门及递归下属员工，跨部门员工不可见。
8. 单据提交后状态实时同步到发起人 ESS（已提交→审批中→通过/驳回）。
9. 审批驳回时驳回意见必填且对发起人可见。
10. 所有 ESS/MSS 操作记入审计日志。

**与其他 Epic 的依赖**：聚合 Epic 01-05；依赖横切的 RBAC 与数据权限。

**Epic 内的硬边界声明**
- 不做移动端原生 App（仅响应式 Web）
- 不做钉钉/企微入口
- 不做消息推送（短信/邮件 MVP 仅留接口骨架）
- 不做员工自助修改身份证/姓名
- 不做经理代发起员工单据（员工自己发起）

## 7. 横切需求

### 7.1 登录认证
- 用户名 + 密码登录；密码 BCrypt 存储
- 密码强度：8-32 位，必须含字母 + 数字（特殊符号可选）
- 连续输错 5 次 → 锁定 30 分钟（解锁靠 Admin 或自动到期）
- Access Token 30 分钟、Refresh Token 7 天
- 首次登录强制改密
- 提供"忘记密码"流程：邮箱或手机验证码重置（接口骨架，发送渠道 MVP 用打印日志模拟）

### 7.2 RBAC 角色权限
- 6 类角色（第 3 章），权限以菜单 + API 路径双层控制
- 同一用户仅能拥有 1 个主角色（MVP 简化）
- Admin 可在角色管理页配置菜单可见性与 API 白名单

### 7.3 组织数据权限
- 员工：仅自己
- Line Manager：本部门 + 所有递归子部门员工
- HR Manager / Admin / Payroll Specialist / Recruiter：全公司（Recruiter 仅候选人 + 公开员工字典）
- 跨部门外派 MVP 不做

### 7.4 操作日志（审计）
- **动作级**：所有写操作（增删改）均记录 who（user）/ when（timestamp）/ what（entity + action）/ result（success/fail + 错误信息），保留 2 年
- **字段级（仅薪资字段）**：薪资档案、工资明细的字段修改额外记录修改前值 + 修改后值
- 仅 Admin 可查询审计日志，提供按用户/实体/时间窗筛选

### 7.5 附件管理
- 存储：本地磁盘 `${HRMS_HOME}/uploads`，按 `年/月/实体类型/实体ID/文件名` 分目录
- 接口抽象：定义 `AttachmentStorage` 接口，本地实现先做，未来切 MinIO 仅替换实现
- 上限：单文件 ≤10MB
- 类型白名单：jpg / png / pdf / docx / xlsx
- 敏感附件加密：身份证扫描件、银行卡扫描件路径在数据库中加密存储（AES）

### 7.6 字典管理
- 集中字典表：假类、合同类型、教育层次、婚姻状况、民族、国籍、币种（仅 CNY）、汇报关系类型等
- Admin 可增删字典条目，所有业务页面引用字典 code 而非硬编码
- 字典变更不做版本化（覆盖即生效）

### 7.7 简化审批流
- 流程引擎：自研轻量审批表（流程节点 / 待办 / 审批历史三张表，不引 Flowable）
- **支持**：最多 3 级串行审批、回退到上一节点、驳回、撤销（发起人在审批中可撤）
- **不支持**：会签、分支、子流程、代理人
- 审批人离职：MVP 简化为"自动顺延到该节点角色的另一在岗人"，无候选人则挂起 + 通知 Admin 介入改派

## 8. 非功能需求

### 8.1 性能（演示规模口径，1 公司/5 部门/20 员工）

| 9 | 多数据库切换 | 停服 → 改 datasource 到 MySQL → 启动 → 同 changelog 自动建表 → 重新登录验证主路径 | 2 分钟 |
| 10 | 一键重置 demo | 执行 `scripts/reset-demo.sh` → 重新登录 → 数据回到初始演示状态 | 1 分钟 |

**总时长**：≈20 分钟（含切换间隙）
